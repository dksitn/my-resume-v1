我的核心邏輯：AI 是助手，我是架構師

「Gemini 合作日誌」是我作為 AI PM 與開發者，如何從零到一建構複雜系統的藍圖。

我的核心哲學是「人機協作，而非單靠 AI」。我（人類）的角色是「PM」與「架構師」，負責提出問題、定義規格、並做出最終決策。

因此，我的第一個貢獻，就是**「設計」我的 AI 助手 (Gemini Gem)**，使其成為一個可控、可預期、具備專業能力的協作夥伴。

這套方法論分為三個層次，由「敏捷 (Agile)」流程貫穿始終：

  1. 層次一：協作框架 (我如何「設計」AI)
  2. 層次二：決策原則 (我如何「驅動」AI 思考)
  3. 層次三：六大執行支柱 (我們如何「協同」實作)

層次一：協作框架 (我如何「設計」AI)

我做的第一件事，就是為 AI 建立一個「作業系統」。我將其設計為一個「多角色、有記憶」的系統，而非一個「什麼都懂、問了才答」的聊天機器人。

以下是我為 AI 設計的核心角色：

角色A：R0 專案狀態核心 (Project State Kernel)

  • [一般人說明] 這就像是 AI 的「專案秘書」或「記憶體」。
  • [我的設計] 我要求 AI 必須在內部維護一個「單一事實來源 (SSOT)」，用 JSON 格式記住專案的完整狀態（例如：`project_spec` 規格、`data_schema` 資料庫綱要、`code_blocks` 程式碼）。
  • [協作成果] 這將 AI 從一個「金魚腦」工具，變成了「有狀態」的專案夥伴。AI 能「記住」我們在 V3.0 決定的資料庫結構，並在我開發 V6.0 時拿出來用，實現了真正的「專案記憶」。

角色B：R-N 角色導向的任務委派 (Role-Oriented Delegation)

  • [一般人說明] 我不讓 AI「通才」回答，而是將其設計為一個「專家團隊」。當我需要時，我會呼叫「特定專家」。
  • [我的設計] 我（PM）不直接對 AI 下令，而是將任務「委派」給我為 AI 設計的「虛擬專家角色」。
  • [協作成果]
    • `R4 (架構導師)` (技術顧問)： 當我們在 V2.0 比較 Zeabur 和 Railway 平台，或在 V4.0 評估是否從 n8n 遷移到 Supabase 時，我呼叫 R4 來提供架構決策分析。
    • `R7 (偵錯專家)` (鑑識科學家)： 當我們遭遇 Bug 時，我呼叫 R7 來執行「RCA 根本原因分析」SOP。
    • `R2 (資安官)` (資安稽核員)： 當我們處理 API Key 或資料權限時，我呼叫 R2 來確保「內建倫理與安全」。
    • `數據分析師` (數據科學家)： 當我們討論 GA/GTM 時，我呼叫此角色來提供數據策略。

層次二：決策原則 (我如何「驅動」AI 思考)

有了這個「專家團隊」，我（人類）就可以運用我的「決策原則」，來驅動 AI 進行高效率的思考與分析。

  • 原則A：數據優先 (Data-First)

    [我的決策] 當「數據可被獲取」時，我們必須獲取它，並確保其「乾淨」。

    [協作成果 (GA/GTM)]

    1. 我 (PM) 的決策：「我必須監測有人進來網站」，這啟動了「數據優先」原則。
    2. AI (數據分析師)： 提供了「安裝 GA/GTM」的技術路徑。
    3. 我 (PM) 的人類洞察： 我立刻提出質疑：「但我換地方換 wifi 不就會 ip 變了?」
    4. AI (數據分析師)： 根據我的質疑，提供了「多層次過濾」的進階解決方案（IP 篩選、GTM 預覽模式、瀏覽器外掛）。
    這是一個「人類決策 -> AI 執行 -> 人類洞察 -> AI 優化」的完美協作。

  • 原則B：費米推理 (Fermi Estimation)

    [我的決策] 當「數據不可被獲取」時（例如預測未來），我會要求 AI 切換至「費米推理模式」，將問題拆解為「邏輯估算」。

    [協作成果 (AI 影片工作流)]

    • 我 (PM) 的需求：「評估 n8n 自動化 AI 影片工作流 (n8n -> ChatGPT -> Veo) 的市場潛力」。
    • AI (費米推理模式)： 我們無法「搜尋」到答案。AI 會在我的指導下，將問題拆解為 `(全球內容創作者數) x (影片製作比例) x (AI 採用率) x (平均願付價格)` 來進行邏輯估算。

層次三：六大執行支柱 (我們如何「協同」實作)

在「決策」做出後，我們（人類 PM + AI 助手）便進入「敏捷實作」階段。這六大支柱是我們在所有專案中都嚴格遵守的工程紀律，以下是更詳細的案例：

  1. 支柱一：架構選型與優化：

    人機協作： 我（PM）提出 V4.0 升級需求，`R4 (AI 導師)` 分析 Supabase 的優勢。我（PM）做出「遷移」的最終決策。

    [更多案例 V2.0] 我（PM）發現 Zeabur 平台的 Bug，`R4 (AI 導師)` 協助分析並確認平台問題，我（PM）決策遷移至 Railway 平台。

    [更多案例 AI 影片] 我（PM）提出「AI 影片流」的需求，`R4 (AI 導師)` 協助分析並選擇了 `n8n` (流程引擎)、`ChatGPT` (劇本) 和 `Veo2` (渲染) 的技術組合。

  2. 支柱二：資料為核 (Data-First)：

    人機協作： 我（PM）設計 `products` 與 `customer_contacts` (V6.1) 的業務邏輯，AI 助手協助生成「關聯式資料表」的 SQL 語法與「外鍵」設定。

    [更多案例 V5.6] 我（PM）提出「需要庫存」的需求，AI 助手協助進行「資料庫綱要演進 (Schema Evolution)」，在 `products` 表上新增了 `quantity` 欄位。

    [更多案例 V5.7] 我（PM）提出「只顯示有庫存的商品」的業務邏輯，AI 助手協助修改了 Supabase 的 `SELECT` 查詢語句。

  3. 支柱三：系統化偵錯 (Systematic Debugging)：

    人機協作： 我（PM）發現 V3.0 「前端讀取失敗」的現象，`R7 (AI 偵錯)` 提供「隔離變因法」SOP。我（PM）執行 SOP，定位到錯誤在 n8n，`R7 (AI 偵錯)` 接著分析並找出 `First Incoming Item` 的根本原因。

    [更多案例 V1.0] 我（PM）遭遇 `Git command not found`，`R7 (AI 偵錯)` 診斷為「環境變數 (PATH) 錯誤」，並提供了修復指南。

    [更多案例 V5.3] 我（PM）遭遇 `products_pkey` 錯誤，`R7 (AI 偵錯)` 診斷為「資料庫主鍵 (Primary Key) 設錯」，RCA 指出是 `name` 被誤設為 PKey，而非 `id`。

  4. 支柱四：內建倫理與安全 (Ethics & Security by Design)：

    人機協作： 我（PM）提出「訪客不能寫入資料」的嚴格要求。AI 助手（`R2 安全官`）提供 Supabase RLS (Row Level Security) 的技術實現方案 (V5.8/V6.1)。

    [更多案例 V2.1] 我（PM）準備串接 Notion API，`R2 (AI 安全官)` 立刻發出「API Key 安全儲存」警告，我（PM）決策將 Key 儲存在 n8n 的 Credential 中，絕不暴露在前端程式碼裡。

  5. 支柱五：本地端開發流程 (Local Development Process)：

    人機協作： 我（PM）在 VS Code 中編寫程式碼，並執行 `Git` 版控。AI 助手（`R0`）則「記住」我的 `code_blocks`。

    [更多案例 V1.0] 我（PM）遭遇 `not a git repository` 錯誤，`R7 (AI 偵錯)` 診斷為「Git 根目錄錯誤」，並指導我（PM）在正確的資料夾下執行指令。

    [更多案例 V1.0] 我（PM）需要在新電腦上工作，AI 助手（`R0`）提供了 `git clone` 與 `git pull` 的SOP，協助我（PM）成功「同步開發環境」。

  6. 支柱六：雲端架構與服務串接 (Cloud Architecture & Integration)：

    人機協作： 我（PM）提出 V7.0「履歷」和「商品」必須分離的「架構解耦」策略。`R4 (AI 導師)` 確認這是「微服務」的最佳實踐，並提供 Railway 上「兩個獨立服務」的部署方案。

    [更多案例 V3.0] 我（PM）要實現「跨網站串接」，AI 助手提供了前端 `fetch()` 程式碼，以「API 呼叫」的方式，讓「履歷網站 (前端)」跨站存取「商品服務 (後端)」的資料庫。

結論：敏捷 (Agile) 是黏合一切的「流程」

「敏捷」是我們的「元流程」，它讓我們能將這套複雜的「人機協作框架」、「決策原則」與「六大執行支柱」拆解成一個個可管理的「衝刺 (Sprint)」。

這套完整的「敏捷流程 + 人機協作框架 + 決策原則 + 六大支柱」方法論，才是我作為 AI PM，能從零到一處理任何複雜專案的真正核心。

痛點 傳統瀑布式開發 (Waterfall) 我的 AI PM 系統方法論
過度開發 試圖一次規劃 V1.0 到 V9.0 的所有功能與完美架構。 Sprint 1 (V1.0) 只做「一個能動的頁面」，我（PM）與 AI 協同驗證「支柱五 (本地流程)」。Sprint 2 (V2.0) 再上後端，驗證「支柱一 (架構選型)」與「支柱六 (雲端架構)」。
SPEC 亂開 開發 V3.0 到一半，又想加入 V8.0 的圖片上傳，導致架構混亂、時程爆炸。 建立「待辦清單 (Backlog)」。有新需求（例如 GA 數據分析），很好，我（PM）將其放入 Backlog，等當前 V9.0 衝刺完成後，再排入下一個 Sprint。絕不干擾當前工作。
