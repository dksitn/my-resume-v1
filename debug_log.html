前言：日誌背後的「方法」

這份「Gemini 合作日誌」不只是一份錯誤修正的流水帳，它是一部實錄，展示了筆者（沈廷翼）作為 AI 專案經理 (PM) 與架構師，如何「設計」並「駕馭」AI 助手（Gemini Gem），以一套有紀律的系統方法論，從零開始建構一個功能完整的全端專案。

這個系統的核心流程是「MVP 優先的架構可行性驗證」——在開發任何功能前，先用最小的成本，驗證所選的「架構」是否行得通。

以下，將以這個方法論（包含「**三大層次**」與「**六大支柱**」）為透鏡，來解讀日誌中的每一個階段。

階段一：V1.0 前端與 Git 管線 (驗證「基礎架構」)

[日誌摘要]：掌握 HTML/CSS/JS 基礎、設定 Git 版控、解決 Git 指令錯誤、並成功在多台電腦同步。

[方法論解讀]

這是整個專案的第一次「MVP 架構可行性驗證」。

在打造任何複雜功能（如購物車）之前，PM 必須先回答一個最根本的問題：「在『本地端』(開發者電腦) 寫的程式碼，有辦法被『雲端』(全世界) 看到嗎？」

這個階段驗證了兩大核心支柱：

  1. 支柱五 (本地端開發流程)： PM 在 VS Code 上撰寫程式碼，並使用 Git 進行版本控制。
  2. 支柱六 (雲端架構)： PM 將 Git 與 Railway 平台串接，成功建立了「CI/CD (持續整合/持續部署)」管線——這意味著只要 PM 在本地端儲存進度 (git push)，網站就會在幾秒鐘內「自動」在雲端更新。

日誌中的 `command not found` 和 `not a git repository` 錯誤，並非「失敗」，而是「系統化偵錯 (支柱三)」的首次實踐。PM 遭遇問題，AI 助手（`R7 偵錯專家`）被呼叫，提供了「環境變數 (PATH) 錯誤」的診斷與SOP。

[本階段結論]：V1.0 架構可行性：驗證通過。「本地 -> 雲端」的自動化管線已建立。

階段二：V2.0/V3.0 後端與平台遷移 (第二次「架構驗證」)

[日誌摘要]：嘗試在 Zeabur 平台部署後端 (n8n+PostgreSQL)，遭遇嚴重連線 Bug。[決策] 遷移至 Railway 平台。

[方法論解讀]

這是**第二次「MVP 架構可行性驗證」**，目標是「後端」。

PM 在「**層次二：決策原則 A (數據優先)**」的指導下，驅動 AI 助手（`R4 架構導師`）進行「工具選型（支柱一）」。最初選擇了 Zeabur 平台。

然而，MVP 測試的最大價值在於「**快速失敗**」。

日誌中的「嚴重連線 Bug」不僅是一個 Bug，它代表著「**此路不通**」。 這是「**系統化偵錯 (支柱三)**」的結果，診斷出這是平台層級的問題，而非程式碼錯誤。

[本階段結論]：V2.0 架構可行性：驗證失敗。 PM 基於 AI 提供的分析，做出「**Pivot (轉向)**」的關鍵決策——果斷放棄 Zeabur，將整個後端架構遷移至更穩定的 Railway 平台。

階段三：資料庫 (PostgreSQL) 實戰 (奠定「資料骨架」)

[日誌摘要]：建立 `categories` 和 `products` 資料表、修正 SQL 拼寫錯誤、成功新增 (INSERT) 與查詢 (SELECT) 資料。

[方法論解讀]

在 V2.0 驗證了 Railway 平台「可行」之後，專案立刻執行「**支柱二 (資料庫結構為核)**」。

這就像蓋房子，在「架構（地基）」穩定後，必須先搭建「**鋼筋骨架（資料庫結構）**」，然後才能蓋牆（開發功能）。

  • 人機協作：PM 定義了「業務邏輯」（例如：一個分類可以對應多個產品）。AI 助手（`R0 核心` 提供了資料庫綱要記憶）則協助生成了 `categories` 和 `products` 的「關聯式資料表」SQL 語法。
  • 偵錯 (支柱三)：日誌中的「SQL 拼寫錯誤」再次證明了「**人機協作**」的必要性。 AI 提供了程式碼，但 PM（人類）作為最終的「品管者 (QA)」，抓住了 AI 造成的錯誤。

[本階段結論]：V3.0 資料庫骨架：建構完成。

階段四 & 五：API 整合 (n8n & Notion) (實踐「安全」與「串接」)

[日誌摘要]：串接 Notion API、(R2 警告)、繞過 n8n 節點 Bug、成功將資料寫入 Notion。

[方法論解讀]

這是對「**支柱四 (內建倫理與安全)**」與「**支柱六 (服務串接)**」的壓力測試。

  • 安全 (支柱四)：在日誌中，`(R2 警告)` 是關鍵。 當 PM 準備串接 Notion API 時，AI 助手（`R2 資安官`）立刻被觸發，發出「**API Key (金鑰) 安全儲存**」警告。
  • 人機協作：PM 接受了 AI 的建議，做出了「**安全優先**」的決策：絕不將金鑰寫死在程式碼中，而是將其儲存在 n8n 的安全憑證庫裡。

[本階段結論]：「安全串接」原則：驗證通過。

階段六：V3.0 動態 API 串接 (「偵錯」的極致展現)

[日誌摘要]：V1.0 前端串接 V3.0 後端失敗。遭遇「讀取商品失敗！」錯誤。日誌詳細記錄了 RCA（根本原因分析）的每一步。

[方法論解讀]

這是整個專案中，對「**支柱三 (系統化偵錯)**」最經典的演繹。這展示了「人機協作」如何解決複雜問題。

  1. PM（人類）：發現「讀取商品失敗！」（**現象**）。
  2. AI（`R7 偵錯專家`）：提供「**隔離變因法**」SOP（**方法**）。
  3. PM（人類）：執行 SOP（使用工具直接呼叫後端 API）。（**執行**）
  4. PM（人類）：回報結果：「後端健康！」（**縮小範圍**）。
  5. PM（人類）：使用 F12 開發者工具，發現 `TypeError: products.forEach is not a function`。（**新線索**）
  6. AI（`R7 偵錯專家`）：分析新線索。AI 指出：這個錯誤意味著「前端」期望收到「**一整箱**商品（陣列）」，但「後端」只給了它「**一個**商品（物件）」。
  7. AI（`R7 偵錯專家`）：追溯到 n8n 節點的 `First Incoming Item` 設定錯誤。（**根本原因**）

[本階段結論]：V3.0 全端串接：成功。 這證明了「人類 PM 的執行力」與「AI 助手的分析力」相結合，是最高效的偵錯模式。

階段七：V4.0 架構升級 (第三次「架構驗證」與「解耦」)

[日誌摘要]：診斷出 n8n 架構的 CORS 根本缺陷。[架構決策] 評估 Supabase 等選項，[V4.0 啟動] 取代 n8n。確立 `my-resume-v1` 和 `puzzles` 兩個獨立服務。

[方法論解讀]

這是**第三次，也是最重要的一次「MVP 架構可行性驗證」**。

  • 架構選型 (支柱一)：PM 診斷出 V3.0 架構的「根本缺陷」（CORS）。
  • 決策 (原則 A)：PM 驅動 `R4 (AI 導師)` 蒐集「正規資料」，對 Supabase、AppWrite 等工具進行了深入的「競品分析」。
  • 人機協作：PM 基於 AI 提供的分析，做出「**再次 Pivot (轉向)**」的重大決策：採用 Supabase (BaaS) 作為 V4.0 的最終後端架構。
  • 雲端架構 (支柱六)：最關鍵的是日誌中的「**確立...兩個獨立的 Railway 服務**」。 這就是「**微服務架構**」的實踐。PM 將「履歷（前端）」和「商品（後端）」**解耦（Decoupling）**，使其可以獨立更新，互不干擾。

[本階段結論]：V4.0 最終架構：驗證通過並部署完成。

階段八 & 九：V5.x / V6.x (在穩定架構上的「敏捷開發」)

[日誌摘要]：實現 CRUD、修復 AI Bug、修復 DB Bug、新增庫存、圖片上傳、修復 RLS Bug、建立訪客下單。

[方法論解讀]

在 V4.0 穩定的「最終架構」之上，專案終於進入了「**敏捷開發 (Agile)**」的功能衝刺階段。

這兩個階段完美展示了所有「六大支柱」的協同運作：

  • 支柱二 (資料庫)：新增 `quantity` (V5.6) 和 `customer_contacts` (V6.1) 欄位，實踐了「資料庫綱要演進」。
  • 支柱四 (安全)：日誌中的 `RLS Bug` (V5.8, V6.1) 正是「**內建倫理與安全**」的證明。 PM 定義了「**訪客 (anon) 不能寫入資料**」的嚴格規則，AI 助手（`R2 資安官`）則提供了在資料庫層級強制執行此規則的 RLS 程式碼。
  • 支柱三 (偵錯)：日誌中甚至抓到了「**由 AI (我) 造成**」的 Bug (V5.0)。 這再次印證了「人機協作」的核心——AI 是助手，PM（人類）必須是最終的品管者。
  • 支柱六 (服務串接)：成功整合了「Supabase Storage（圖片上傳）」(V5.8)，為架構添加了新的雲端服務。

[本階段結論]：V6.2 成功！ 專案具備了「安全」、「可擴展」、「資料驅動」的所有特性，完整的功能成功交付。
